What's built today

The platform has two main surfaces — a public side for musicians and an authenticated side for organisations — plus a built-in admin panel you (Nathan) control.

For musicians (no account needed)

  • Map-first search. Opens on a map of the UK, drops a marker for each opportunity. Click a marker for a quick popup; click a card in the sidebar to open the full detail page.
  • Filters: free-text search (title, description, instrument), type (Classes / Ensemble / Workshop / Performance / Lessons / Project), postcode or place name for proximity, radius picker (5–100 km).
  • 1.79 million UK postcodes stored locally in the platform, so "near me" works instantly with no external geocoding service.
  • Full detail page per opportunity: description, location, age group, instruments, linked organisation with a verified badge.
  • Advice page stub — placeholder copy, ready for real content.

For organisations (after admin verification)

  • Register, log in, log out. Standard email + password flow.
  • Dashboard showing opportunities posted by the organisation, with the verified / pending badge next to the org name.
  • Until the admin flips them to "verified", new organisations can see their dashboard but cannot publish opportunities — a deliberate safeguarding gate.

Architecture in plain English

Three pieces cooperate on a single small server. A helpful way to picture it:

The shop window
Website
What visitors see in their browser. Plain static files (HTML, CSS, pictures) so it loads fast anywhere.
The filing cabinet
Database
Holds every opportunity, every organisation, every postcode. One file on disk. Instant to read, easy to back up.
The receptionist
Reverse proxy
Handles HTTPS, routes each visitor's request to the right place, serves files efficiently.

The “filing cabinet” is PocketBase — an open-source single-file application. The “shop window” is built with SvelteKit and compiles down to a folder of static files. The “receptionist” is Caddy, which also handles free HTTPS certificates automatically.

All three live on the same small machine. No Docker, no Kubernetes, no managed cloud service, no database server. If the server disappeared, you could copy one folder to a new machine and be running again in minutes.

The admin side

PocketBase ships with a built-in admin panel. Opening it in a browser gives you (Nathan) a spreadsheet-like view of every record on the platform — with search, filters, sort, inline edit, CSV export, and CSV import.

This is where you:

  • Review new organisations and flip the verified switch once you're satisfied (email, phone check, DBS / charity registration, however you choose to confirm).
  • Export any table as CSV for reporting.
  • Import a CSV of opportunities directly — useful if someone sends a spreadsheet instead of using the form.
  • Edit anything manually if a user gets stuck.

No custom admin UI was needed — it's a free side-effect of the technology choice.

What's included for that monthly fee:

  • Full HTTPS (green padlock in the browser) — Caddy gets a free certificate from Let's Encrypt and renews it automatically.

How many users can this handle?

A realistic answer for the UK Music Experiences use case:

~5,000simultaneously active users — comfortable on the cheapest VPS
~100,000+daily visits in a typical day — easy headroom
< 1 GBmemory used at idle — 3+ GB left over
~1 msaverage database read on an indexed query

For context: the platform will most likely see a few hundred users per day in its first year, with spikes during open-call announcements. The server will be more than 90 % idle the vast majority of the time.

The honest bottleneck on growth is not the server — it's your ability to manually review and verify new organisations. One admin is the gating factor, not the hardware. The day you need a second admin to keep up with registrations is the day the platform has succeeded.

Appendix: the tech stack (one page)

For completeness, in case you share this report with a developer:

FrontendSvelteKit compiled to static HTML / CSS / JavaScript
Map libraryLeaflet 1.9 with CartoDB tiles (no API key, free)
BackendPocketBase 0.26 — a single Go executable providing database, auth, REST API, file storage, and admin UI
DatabaseSQLite (inside PocketBase). One file. Fully backed up by copying that file.
Reverse proxyCaddy 2.11 — static file serving, automatic HTTPS, URL rewriting
Geocoding dataONS Postcode Directory + OS Open Names — UK government open data, stored locally
HostingAny Linux VPS; Hostinger KVM 1 or Hetzner CX22 tested
Runtime cost€4.99/month VPS + ~€10–15/year domain
External dependenciesNone at runtime. No map API, no geocoding API, no database service.
Ops surfaceTwo long-running processes (PocketBase + Caddy). Both managed by systemd. Updates = drop-in binary replacement.

The engineering principle throughout has been to remove pieces, not add them. Every component that doesn't earn its keep — a second database server, a Node.js runtime, Docker, a JavaScript build server, a managed search service — was cut. What remains is what you cannot do without.
